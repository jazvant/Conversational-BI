# Technical Decisions

Key decisions made during design and build, with rationale. Written for judges and contributors who want to understand why the system is built the way it is.

---

## Data layer

### DuckDB over pandas for query execution

pandas loads entire tables into RAM as DataFrames. `order_products_prior` has 32 million rows — loading it into pandas consumes approximately 3–4GB of RAM and makes every subsequent operation slow. DuckDB reads CSVs natively, executes queries using columnar parallel scans, and fits the entire working set in well under 1GB. For a BI tool where every query is a read, DuckDB is the right engine.

DuckDB also eliminates an ETL step — `read_csv_auto()` registers a CSV as a queryable table in one line with automatic type inference.

### Normalised tables over a single flat table

Flattening all 6 tables into one would store department names like `"beverages"` 2.1 million times instead of once. The resulting table would be 15–25GB and provide no query performance benefit — DuckDB's join performance on 21-row and 134-row dimension tables is essentially free since they fit entirely in cache.

The approach taken is to store normalised base tables as the source of truth and create a `order_details` view that pre-joins order and product context. All LLM-generated SQL queries `order_details` for transactional queries and joins to `products`, `aisles`, and `departments` when product hierarchy is needed.

### `order_details` as the primary query surface

The Instacart dataset ships with `order_products__prior` and `order_products__train` as separate files — a competition artefact, not a meaningful business boundary. For BI purposes, combining them into a single `order_details` view (UNION ALL + join to orders) gives analysts a complete purchase history without needing to think about the split. The `eval_set` column is preserved for queries that specifically need to distinguish prior from train.

## Schema layer

### Schema context generated programmatically, not written manually

`schema_context.txt` — the LLM's only view of the database — is generated by `m2_run.py` from the live database rather than written by hand. This means it stays accurate as the database evolves. Manually written schema prompts drift from reality; generated ones cannot.

Sample values are fetched live from DuckDB rather than hardcoded. This matters for columns like `order_dow` where the encoding (0=Saturday, not 0=Sunday) is non-obvious from the column name alone.

### Query rules injected as explicit constraints, not relied on from training

The LLM knows the Instacart dataset from training data — it would likely generate correct `order_dow` encoding even without being told. We inject explicit rules anyway because:

1. Training knowledge is probabilistic and inconsistent across phrasings
2. The rules make the system's behaviour predictable and auditable
3. When rules are explicit in the prompt, failures are debuggable

Every rule in the query rules section exists because without it the LLM made a specific, repeatable mistake during M2.4 prompt validation.

## SQL generation

### Free text for SQL, not structured output

SQL cannot be schema-constrained. Unlike intent classification or narrative generation where the output is a small set of known fields, SQL is open-ended — any valid DuckDB query is a correct output. Forcing SQL through a Pydantic schema would require defining a schema that covers all of SQL syntax, which is not feasible.

The approach is to strip markdown fences defensively on the raw text output and pass it directly to the executor.

### Structured output with Pydantic for all other LLM calls

Every LLM call that returns structured data — intent classification (Planner), narrative generation (Critic) — uses the Anthropic tool_use API with `tool_choice` forced. This guarantees the response matches the Pydantic schema. No JSON parsing, no string splitting, no silent fallbacks on malformed output.

The schema files live in `m5_schemas.py` only. No schema is defined inline in any other file — this ensures changes propagate consistently and the schemas can be tested independently of the API.

### Pydantic field validators enforce business rules

Rather than checking constraints after construction, validators run at model creation time:

- `CriticOutput.followup` always ends with `?` — enforced by validator, not prompt instruction
- `PlannerOutput.subqueries` is always empty for non-multistep intents — enforced by validator
- `BenchmarkQuestionResult.latency_ms` cannot be negative — enforced by `ge=0` constraint

This means a malformed LLM response that passes the schema type check but violates a business rule raises a `ValidationError` immediately rather than silently producing bad data downstream.

## Execution safety

### Read-only DuckDB connection as the primary security layer

Opening DuckDB with `read_only=True` is enforced at the driver level. Any write operation — regardless of how the SQL was generated or what the application code does — is rejected by DuckDB before it reaches the storage engine. This cannot be bypassed by application code.

The SQL keyword blocklist (`m7_input_validator.py`) is a secondary layer that provides better user-facing error messages. Both are needed:

- The keyword blocklist catches destructive intent before the database sees the SQL, surfacing a clear message to the user
- The read-only connection catches anything that slips through, including SQL generated in unexpected ways

### 15-second query timeout via threading

DuckDB does not support query cancellation from the same thread. The threading approach runs the query in a daemon thread and lets the main thread move on after the timeout. The daemon flag ensures the thread does not block application shutdown. The tradeoff is the query continues running in the background until DuckDB terminates it via the memory limit — acceptable for a BI tool where the alternative is a hung UI.

15 seconds was chosen as a balance between allowing complex multi-table aggregations on the 32M-row table (which can take 8–12 seconds) and keeping the interactive UI responsive.

### Error message sanitisation before UI display

DuckDB error messages include internal file paths, line numbers, and exception class names that leak implementation details. `m8_output_sanitiser.py` intercepts all errors before they reach the UI and replaces them with either a safe descriptive substring (for user-actionable errors like syntax errors or missing columns) or a generic message (for internal errors that the user cannot act on). The raw error is logged at DEBUG level for internal debugging.

## Conversational memory

### Programmatic result summarisation over LLM summarisation

After each query, the result dataframe is summarised into a compact text string and stored in conversation history alongside the SQL. This summary is what enables follow-up questions like "why is produce ranked highest?" — the LLM can see the actual values from the prior result.

Programmatic summarisation (inspecting the dataframe with Python logic) was chosen over a second LLM API call for three reasons:

1. Zero additional latency and cost per query
2. No hallucination risk — the summary only describes what is actually in the dataframe
3. For the follow-up patterns most commonly needed (referencing specific values, comparing ranks), programmatic summaries are sufficient

The interface (`summarise_result(df, question, sql) -> str`) is designed so Architecture 2 can swap in an LLM-based summariser later without changing callers.

### Dynamic history window expansion for multi-step questions

The default conversation window is 3 turns (6 messages). When the current question contains multi-step keywords (`compare`, `first`, `then`, `versus`), the window expands to 5 turns automatically. This avoids the cost of a large context window on every query while providing the extended context that multi-step reasoning genuinely needs.

## Architecture 2

### Planner as a classifier, not a SQL generator

The Planner's only job is to classify intent and route — it does not generate SQL, plan joins, or decompose queries itself. This keeps the Planner prompt small and focused, which produces more reliable classification. SQL generation remains entirely in M3 where the schema context and retry logic already live.

The four intent categories (data_query, conversational, multistep, cannot_answer) were chosen because they represent meaningfully different handling paths, not because they are exhaustive. A question that doesn't fit cleanly defaults to `data_query`, which is always safe — the worst outcome is an unnecessary SQL query.

### Critic generates narrative only for non-scalar results

Generating a narrative for "How many orders are there? → 3,421,083" adds no value. The Critic skips narrative generation when the result has one row or fewer (`NARRATIVE_MIN_ROWS = 2`). For multi-row results, the narrative provides genuine value by identifying the most interesting finding, flagging data quality issues, and suggesting the next analytical question.

### `ARCHITECTURE` flag as the kill switch

Setting `ARCHITECTURE = 1` in `config.py` restores exact Architecture 1 behaviour with no other code changes. This enables A/B comparison and provides a safe rollback path if the Planner causes regressions on simple queries (the extra API call adds approximately 1–2 seconds of latency per question).

## Evaluation

### Three independent scores per benchmark question

A single pass/fail score per question hides important information. A question where the SQL is correct but the result values fall outside expected ranges is a fundamentally different failure from one where the SQL references a non-existent column. The three-score model (SQL correctness, DB execution, result sanity) makes these distinct.

SQL correctness is a keyword check — reliable and fast. DB execution is binary — the query either runs or it doesn't. Result sanity checks values against expected ranges — the most informative but also the most likely to have false negatives when expected ranges are set conservatively.

### Benchmark routes directly to M3, bypassing the Planner

The current benchmark measures SQL generation quality, not classification quality. This was a deliberate choice — measuring classification accuracy requires a different evaluation harness with labelled intent ground truth. The benchmark as implemented gives a stable, reproducible measure of the core SQL pipeline that can be compared across architecture changes. A future enhancement would add a separate classification accuracy benchmark.
